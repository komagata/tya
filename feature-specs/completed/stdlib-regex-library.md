# Feature: stdlib Regex Library

## Goal

Add a v1.0.0 standard-library regular expression package for validation,
search, splitting, and replacement without adding regex syntax to the language.

## Context

Tya v1.0.0 intentionally keeps the language syntax small. String methods cover
basic text operations such as `contains`, `starts_with`, `ends_with`,
`replace`, `split`, `lines`, and `chars`, but general pattern matching is a
standard-library concern. A v1-ready scripting and systems language needs a
portable regex API for command-line tools, log processing, data validation, and
text transformation.

The accepted direction is a class-style `regex/Regex` package backed by the
compiled runtime. Regexes are values, invalid patterns raise structured errors,
and matching is explicit through methods rather than operators or literals.

## Behavior

- `import regex` exposes `regex.Regex`.
- `Regex.compile(pattern, options = {})` compiles a regex and returns a regex
  value.
  - `pattern` must be a string.
  - `options` must be a dictionary.
  - Supported options:
    - `ignore_case`: bool, default `false`;
    - `multi_line`: bool, default `false`;
    - `dot_all`: bool, default `false`.
  - Unknown option keys are invalid.
  - Invalid patterns raise a structured error with `kind: "regex"` and a
    stable `code`.
- `Regex.match?(pattern, text, options = {})` returns a bool for full-pattern
  or search semantics as documented below.
- `Regex.search(pattern, text, options = {})` returns the first match
  dictionary or `nil`.
- A compiled regex supports:
  - `match?(text)`: bool, true when the pattern appears anywhere in `text`;
  - `find(text)`: first match dictionary or `nil`;
  - `find_all(text)`: array of match dictionaries in left-to-right order;
  - `split(text, limit = nil)`: array of strings;
  - `replace(text, replacement, limit = nil)`: string.
- Match dictionaries contain:
  - `text`: matched text;
  - `start`: rune index of the match start;
  - `end`: rune index immediately after the match;
  - `groups`: array of captured group strings or `nil` for unmatched optional
    groups.
- Matching uses UTF-8 text semantics.
  - Indexes are Unicode rune indexes, matching Tya string indexing.
  - Invalid UTF-8 text input is rejected before matching.
- Replacement supports numeric capture references with the explicit `${1}`
  spelling.
  - `$$` emits a literal dollar sign.
  - Unknown capture references are invalid.
- Regex syntax is runtime-backed and portable across supported release
  platforms.
  - The v1 contract must document the accepted syntax subset.
  - Lookbehind, backtracking-control verbs, engine-specific extensions, and
    locale-dependent matching are not part of v1.0.0 unless explicitly
    documented by the accepted runtime backend.

## Scope

- `lib/regex/Regex.tya`
- runtime-backed regex intrinsics for interpreter and generated C
- `docs/SPEC.md`
- `docs/STRICT_SEMANTICS.md`
- stdlib docs generated by `tya doc`
- parser/checker only where needed for import and diagnostic coverage
- tests under `tests/` and runtime/codegen tests for parity

## Out of Scope

- Regex literal syntax.
- Pattern matching syntax changes.
- Operator overloads for regex matching.
- Streaming regex APIs.
- Locale-aware collation or locale-dependent character classes.
- Engine-specific syntax beyond the documented v1 subset.

## Acceptance Criteria

- `regex.Regex` is documented in `docs/SPEC.md` as part of the v1.0.0 stdlib
  surface.
- Valid regex patterns can be compiled and reused.
- Invalid regex patterns raise structured errors with stable diagnostic codes.
- `match?`, `find`, `find_all`, `split`, and `replace` work in both the
  interpreter and generated C.
- Match indexes are rune indexes, not byte offsets.
- Replacement capture references are deterministic and reject unknown captures.
- Unsupported options and wrong argument kinds raise structured errors.
- Existing self-host fixed-point gates remain valid.

## Tests To Add

Eval/runtime tests:

- `TestRunRegexCompileAndSearch`
  - Compiles a pattern and searches text.
  - Expected: first match dictionary contains `text`, rune `start`, rune `end`,
    and captured `groups`.

- `TestRunRegexFindAllSplitReplace`
  - Exercises `find_all`, `split`, and `replace` with capture references.
  - Expected: deterministic left-to-right results.

- `TestRunRegexUtf8RuneIndexes`
  - Matches text containing multi-byte UTF-8 characters.
  - Expected: `start` and `end` are rune indexes.

- `TestRunRegexInvalidPatternsAndOptionsRaiseStructuredErrors`
  - Invalid pattern, unknown option, wrong option kind, and unknown replacement
    capture.
  - Expected: structured regex errors with stable codes.

Codegen tests:

- `TestEmitCRegexProgram`
  - Builds and runs a program using compile, find, split, and replace.
  - Expected: generated C behavior matches interpreter behavior.

Testscript coverage:

- `v1_stdlib_regex.txtar`
  - Covers CLI-level `tya run` and `tya build` behavior for valid and invalid
    regex usage.

Documentation tests:

- `TestSpecDocumentsRegexStdlib`
  - Expected: `docs/SPEC.md` lists `regex/Regex` and documents the accepted
    v1 regex syntax subset.
